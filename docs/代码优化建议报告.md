# Smart Extractor 代码深度优化与重构建议报告

经过对项目核心调度 (`task_manager.py`)、流程编排 (`pipeline.py`)、底层抓取 (`fetcher/playwright.py`) 以及大模型提取 (`extractor/llm_extractor.py`) 的扫描与分析，目前项目的代码结构非常清晰，模块解耦做得很优秀。但在**高并发、大吞吐量和资源利用率**上，还有以下几个非常关键的优化点：

## 1. 核心并发模型：从多线程转向原生 Asyncio (⭐️⭐️⭐️⭐️⭐️)
**现状**：在 `pipeline.py` 的 `run_batch` 和 `task_manager.py` 中，使用了 `ThreadPoolExecutor` 来进行并发控制。
**痛点**：网页抓取和 LLM API 调用本质上都是 **I/O 密集型操作**。使用 OS 级别的线程池（如 50 个线程）会带来巨大的内存开销和上下文切换成本，且受限于 Python 的 GIL。
**优化建议**：
- 全面引入 `asyncio`。
- 将 Playwright 切换为 `playwright.async_api`。
- 将 LLM 调用从同步 `client.chat.completions.create` 切换为 `AsyncOpenAI` 客户端。
- 使用 `asyncio.gather` 或 `asyncio.Semaphore` 控制并发，在极大降低 CPU 与内存负担的情况下，轻松支撑千级规模的并发请求。

## 2. Playwright 浏览器上下文上下文池 (Context Pool) (⭐️⭐️⭐️⭐️)
**现状**：`PlaywrightFetcher._create_page()` 中，每次抓取新页面都会 `browser.new_context()` 然后新建 page，抓完再销毁 page 和 context。
**痛点**：对于海量同站点的抓取，频繁创建 Context 开销极大。
**优化建议**：
- 引入 **Context Pool (上下文池)** 机制。为不需要极致隔离的批量同源任务复用 `BrowserContext`，只需 `context.new_page()` 即可。
- 对于确实需要防关联的任务，再使用全新 Context 与随机 User-Agent。这能让动态抓取速度提升至少 30%。

## 3. LLM 提取的批量并行盲点 (⭐️⭐️⭐️⭐️)
**现状**：在 `llm_extractor.py` 的 `extract_batch` 方法中：
```python
for i, text in enumerate(texts):
    result = self.extract(text, schema)
```
**痛点**：这是一个完全串行的 `for` 循环！如果一次性传入 100 篇文章给 LLM 提取，它排队一条条等待 API 返回，将造成极其严重的堵塞。
**优化建议**：
- 将内部的串行循环改为并发派发（即便外部不用 Async，这里也必须用线程池包裹，或者结合第 1 点直接 `await asyncio.gather(*tasks)`），让 LLM API 并发响应，耗时将从 `N * T` 降到近乎 `T`。

## 4. 缺失大文本截断与 Token 管理 (⭐️⭐️⭐️)
**现状**：直接将 `cleaned_text` 丢进 prompt `template.format(text=text)`。
**痛点**：如果遇到超长博文或没洗干净的冗长 DOM，文本轻松突破 LLM 的 `max_tokens` 或上下文窗口上限，导致整条 API 报错失败（且浪费请求）。
**优化建议**：
- 在 `llm_extractor.py` 中调用 `tiktoken` 库在发起 HTTP 请求前进行廉价的本地 Token 估算。
- 如果超出模型限制，执行字符串截断并记录 Warning，而不是强行发给 API 导致失败。

## 5. 内存级任务调度的脆弱性 (⭐️⭐️⭐️)
**现状**：`task_manager.py` 将所有调度任务存在内存 `self._tasks: list[Task]` 中。
**痛点**：当您用 CLI 提交了 1 万个 URL 的 `batch`，抓到 5000 个时如果遭遇断电或进程 OOM 崩溃，所有原本在内存中的排队进度和失败记录全部丢失。
**优化建议**：
- 为 TaskManager 引入 **持久化检查点 (Checkpointing)** 机制。可以借助现有的 `SQLiteTaskStore`（目前主要在 web 端用），将 CLI 批处理的状态也实时写入 SQLite。
- 崩溃重启后，只需读取 DB 并仅恢复 `status="pending"` 的任务，实现真正的稳健断点续传。

---
**总结**：系统的上层抽象极其到位，但底层 I/O 仍受限于同步阻塞与线程模型。建议下一步优先实施 **第 1、3 点（并发模型切换）**，能立竿见影感受到抓取和解析速度的百倍提升。
